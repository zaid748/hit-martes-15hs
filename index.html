    Git Desarrollo colaborativo

    sistema de control de versiones

    Areas de Git

    Working Directory

    El area de trabajo hace referencia a la carpeta donde creamos el repositorio. muy rara vez utilizamos la linea de comandos en esta zona, pero es donde realizaremos todas las modificaciones relacionadas con nuestro proyecto

    • dir || ls muestra el contenido del directorio actual
    • mkdir <dir> crea un directorio con el nombre especificado
    • cd <dir> cambia el directorio actual
    • git init inicializa un repositorio de git
    • rm <dir> elimina un archivo, pero podemos especificar que elimina una carpeta, por ejemplo: rm -rf .git

    Staging Area (INDEX)

    El area de control de cambios corresponde al archivo que se encuentran en la carpeta .git que se crea al iniciar un repositorio. Aqui se regitran los archivos a los que le vamos a hacer seguimiento y es el punto de comparacion para confirmar dichos cambios. Es necesario agregar los archivos con los que estemos trabajando a dicha area para avanzar con el historial de cambios.

    • git status compara el working directory con el contenido del staging area(INDEX) y mustra en pantalla el estado de los archivos del mismo
      • untrecked El archivo no se encuentra en el index por lo tanto no se le esta haciendo seguimiento
      • tracked Los archivos/cambios fueron agregados al INDEX y estan para ser confirmados
      • modified El archivo que se encuentra en el working directory difiere con los cambios almacenados en el INDEX
      • commited Los cambios fueron confirmados y almacenados en el repositorio
    • git add <file> agrega al INDEX los cambios/archivos especificados, para agregar todo el contenido de un directorio podemos utilizar el comando git add
    • git restore <file> Restablece los cambios en el working directory, a la ultima version confirmada del archivo
    • git reset <file> --cached Quita del INDEX el archivo seleccionado, pero mantiene la copia en el Working Directory

    Repository (Local/Remote)

    Ele repositorio corresponde a la base de datos donde se almacena los cambios en formaro BLOB(Binary Object File) y es el lugar al que pademos acceder cada vez que necesitemos revisar algun estado anterior de nuestro proyecto. Esto es posible gracias a los commit

    • git commit Abre el editor predeterminadi para generar una confirmacion de cambios, podemos utilizar comandos como git commit -am "msg" para agregar los cambios de los archivos con seguimiento al INDEX y generar un mensaje de commit
    • git diff <commit> compara el working directory con la referencia especificado, esta puede ser relativa ej: HEAD^(ancestro) o HEAD~2(anteriores) o absoluta utilizando el numero de HASH
    • git blame <fiel> Muestra el archivo seleccionado con el usuario, fecha y hora en las que se realizaron modificaciones en el mismo. En caso de haber cambios sin confirmacion muestra la leyenda not commited yet
    • git checkot <commit> Permite que navegemos en el historial de cambios para ver las versiones de nustro proyecto
